Issue tracking override: how does per-repo relate to global?

The global issue tracking config is set in Settings. Each repo can override it. How?

A

Toggle: "Use global" / "Custom for this repo"

By default the section shows "Using global issue tracking settings" with a summary of the inherited values. A toggle switches to custom mode, where the repo gets its own copy of all fields (pre-filled from global). Changes only affect this repo.

Default state (inheriting)
Issue Tracking
Custom
Inheriting global: enabled, exclusive, develop=[enhancement, bug], skip=[wontfix]

Develop labels
Review-only labels

Pros

  • Clear — you see exactly what's inherited vs custom
  • Can't accidentally break global config from a repo page
  • Pre-fills from global = easy to start customizing

Cons

  • All-or-nothing — can't override just one label list
B

Per-field override (each field has its own "inherit/custom" toggle)

Each label list has a small toggle: inherit from global or set custom. Allows overriding just develop_labels while keeping global skip_labels.

Pros

  • Granular — override only what's different

Cons

  • Complex UI — 8 toggles in the issue tracking section alone
  • Hard to see at a glance what's inherited vs custom
  • Daemon config doesn't support partial override (it's all-or-nothing per repo)
